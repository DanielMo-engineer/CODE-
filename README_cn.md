AI 长者看护系统
——基于 RDK X5 边缘计算平台的多模态智能摔倒检测与告警系统

摘要
随着我国人口老龄化程度加深，独居老人的居家安全问题日益突出，摔倒作为威胁老年人生命健康的首要风险因素，亟需智能化、低延迟的监测手段。本作品设计并实现了一套基于 RDK X5 边缘计算平台的 AI 长者看护系统，面向独居老人及儿童的居家安全监护需求，提供全天候、多维度的智能看护解决方案。
系统以 RDK X5 开发板为核心硬件平台，搭载双核 BPU 神经网络处理器，采用 YOLO11n-pose 单模型实现人体检测与 17 个 COCO 关键点姿态估计的同步输出，经 INT8 量化后在边缘端实现实时推理。针对摔倒检测任务，本系统创新性地提出”三级过滤”策略：第一级基于人体框宽高比判定水平姿态，第二级计算垂直下落速度识别快速倒下动作，第三级通过 17 个关键点计算人体倾角与水平度进行姿态验证，并引入”三选二”投票机制与连续帧持久化过滤，有效降低误报率。在告警层面，系统支持 HDMI 本地显示、Web MJPEG 视频流（端口 8080）、Server 酱微信推送等多渠道告警，确保危险事件第一时间触达监护人。
此外，系统还集成了基于面部关键点的 7 类表情识别模块、Flask Web 远程监控平台（支持 PWA 离线访问与 mDNS 本地域名解析）以及 DeepSeek 云端 AI 对话功能，构建了”边缘实时推理 + 云端智能交互”的混合架构。经实测，系统在 480p 分辨率下检测帧率达 8 FPS，BPU 算力占用约 10 TOPS，端到端检测延迟低于 50 ms，满足实时看护需求。本作品具有部署便捷、成本可控、隐私安全等优势，适用于家庭、养老院、医院病房等多种场景，对提升居家养老安全水平具有重要的现实意义与应用价值。

第一部分 作品概述
1.1 功能与特性
本系统围绕独居老人居家安全的核心痛点，构建了覆盖”感知—分析—决策—告警—交互”全链条的智能看护能力。在感知层，系统支持 USB 摄像头、MIPI CSI 接口摄像头及 GS130W 等多种摄像头方案自适应接入，通过 NV12 格式帧采集实现视频输入。在分析层，基于 YOLO11n-pose 模型同步输出人体检测框与 17 个骨骼关键点，结合三级过滤算法实现高精度摔倒检测；同时集成表情识别模块，可识别平静、高兴、悲伤、惊讶、愤怒、恐惧、厌恶 7 种情绪，辅助判断被看护人精神状态。在决策与告警层，当检测到摔倒事件时，系统通过蜂鸣器本地告警、微信推送、语音电话，Web 弹窗等多渠道同步通知监护人。在交互层，提供 Flask Web 可视化平台，支持实时视频查看、告警记录查询、系统状态监控、AI 对话及心情日志功能，并支持 PWA 离线缓存与 mDNS 本地域名访问，无需记忆 IP 地址即可便捷访问。

1.2 应用领域
本系统主要面向以下应用场景：
（1）独居老人居家看护：针对我国数量庞大的独居老年群体，系统可 7×24 小时不间断监测室内活动状态，在老人摔倒、突发疾病倒地等紧急情况下秒级告警，为”空巢老人”提供安全兜底。
（2）儿童安全监护：部署于家庭客厅、儿童房等区域，实时监测儿童活动姿态，识别跌倒、攀爬等危险行为，及时通知家长干预，降低意外伤害风险。
（3）养老机构集中管理：在养老院、日间照料中心等场所批量部署，通过 Web 平台实现多设备集中监控与告警统一管理，提升护理人员工作效率，降低巡护盲区。
（4）医院病房术后监护：针对术后恢复患者或行动不便病人，提供非接触式姿态监测，在患者跌倒或试图离床时及时通知医护人员，补充传统呼叫系统的响应空白。

1.3 主要技术特点
（1）单模型端到端姿态估计与检测：采用 YOLO11n-pose 单模型同时输出人体检测框与 17 个 COCO 关键点，相比”检测 + 姿态双模型”方案，推理延迟降低约 40%，内存占用减少约 30%，且骨架可视化数据直接来源于模型原生输出。
（2）BPU 硬件加速与 INT8 量化：充分利用 RDK X5 双核 BPU 的 10 TOPS INT8 算力，对 YOLO 模型进行量化部署，实现边缘端低功耗、低延迟的实时推理，无需依赖云端算力，保障数据隐私。
（3）三级过滤摔倒检测算法：融合框形状分析、垂直速度计算、人体倾角验证三个维度，配合”三选二”投票与连续 N 帧持久化确认机制，在保持高检出率的同时显著抑制误报。
（4）边缘-云端混合架构：边缘端负责实时视频推理、摔倒检测、表情识别等低延迟任务；云端 DeepSeek API 承担 AI 对话、心情日志总结等非实时智能交互任务，二者协同实现性能与体验的最优平衡。
（5）无代理直连视频流与 PWA 支持：浏览器直连摄像头 MJPEG 流服务器，避免 Flask 代理阻塞；前端采用 PWA 技术实现 Service Worker 离线缓存，断网时仍可查看历史记录与离线页面。

1.5 主要创新点
1.单模型融合架构创新：突破传统”先检测后姿态”的级联范式，以 YOLO11n-pose 单模型实现检测与姿态估计一体化，在边缘算力受限场景下显著提升推理效率。
2.多维度摔倒判别机制：首创”宽高比 + 垂直速度 + 人体倾角”三级过滤策略，结合关键点有效性校验（≤10 个有效关键点时禁止判定），大幅提升复杂场景下的判别可靠性。
3.多渠道异构告警体系：集成本地蜂鸣器、微信推送及 Web 弹窗多种异构告警通道，适配不同网络环境与用户习惯，确保告警必达。
4.无代理视频流传输优化：针对 Flask 单线程代理阻塞问题，采用浏览器直连摄像头 HTTP 服务器的架构设计，从根本上解决视频流与 API 请求的并发冲突。
1.6 设计流程
本系统的设计遵循”需求驱动、分层迭代”的原则，整体流程如下：
需求分析 → 硬件选型 → 模型训练与量化 → 核心算法开发 → 软件系统集成 → 联调测试与优化
首先，通过调研独居老人看护场景的核心需求，确定以实时摔倒检测为主、表情识别与远程交互为辅的功能定位。其次，选定 RDK X5 作为边缘计算平台，完成 USB/MIPI 摄像头、蜂鸣器等外设的硬件适配。随后，将 YOLO11n-pose 模型转换为 BPU 支持的 INT8 格式并部署验证。在此基础上，开发三级过滤摔倒检测算法、多渠道告警模块及 Flask Web 平台。最后，进行软硬件联调，针对误报率、延迟、稳定性等指标进行多轮迭代优化，形成最终作品。

第二部分 系统组成及功能说明
2.1 整体介绍
本系统采用分层模块化架构，整体可分为感知层、边缘计算层、网络服务层、应用交互层四个层次，各层之间通过标准化接口进行数据交互。系统整体框图如图 2-1 所示。

感知层由摄像头模组（USB/GS130W/MIPI）与音频采集设备组成，负责原始视频帧与音频信号的采集。边缘计算层以 RDK X5 为核心，运行 BPU 加速的 YOLO11n-pose 推理引擎、三级过滤摔倒检测算法、表情识别模块及本地告警驱动（蜂鸣器）。网络服务层包括 Flask API 服务器（端口 5050）、摄像头 MJPEG 推流服务器（端口 8080）及 HTTP 独立推流服务，提供 RESTful API、SSE 事件流与视频流服务。应用交互层涵盖 Web 前端（PWA）、微信推送通道及 DeepSeek 云端 AI 对话接口，实现人机交互与远程通知。
数据流向为：摄像头采集 NV12 帧 → BPU 推理获取人体框与关键点 → 三级过滤算法判定摔倒 → 触发本地告警与网络通知 → Web 前端实时展示与历史记录存储。表情识别与 AI 对话作为独立支线，分别基于面部 ROI 与文本接口并行运行。
2.2 硬件系统介绍
2.2.1 硬件整体介绍
系统硬件以 D-Robotics RDK X5 开发板为核心控制单元，外围连接视频采集、音频输出、网络通信及本地告警四大类模块。RDK X5 搭载 8 核 ARM Cortex-A 处理器与双核 BPU 神经网络处理器，配备 8GB LPDDR4 内存与 64GB eMMC 存储，集成千兆以太网、WiFi 6 及 Bluetooth 5 通信能力，为边缘 AI 推理提供充足的算力与连接性支撑。
硬件整体连接关系如下：RDK X5 通过 USB 接口或 MIPI CSI 接口连接摄像头模组；通过 GPIO 连接蜂鸣器与GSM 模块的 UART 控制线；通过 HDMI 接口连接本地显示设备；通过网络接口接入局域网，实现与云端服务器及移动终端的通信。
2.2.2 机械设计介绍
本系统以桌面/墙面灵活部署为设计目标，整体采用模块化堆叠结构。RDK X5 开发板固定于散热底座上，通过柱支撑形成上下双层结构：上层为 RDK X5 主板及散热片，下层为GSM 模块、蜂鸣器驱动板及电源管理模块。摄像头通过可调节支架固定，支持俯仰角度手动调节，以适应不同房间布局与监控视角需求。外壳采用ABS材质封装，预留摄像头视窗、散热孔及蜂鸣器出声孔，整体尺寸控制在 15 cm × 12 cm × 8 cm 以内，便于家庭环境隐蔽部署。

2.2.3 电路各模块介绍
（1）核心主控模块
RDK X5 开发板为系统核心，关键信号包括： - 输入：5V/3A 直流电源输入、USB 摄像头数据输入、MIPI CSI 差分信号输入、以太网 RJ45 接口、UART 串口（用于通信）。 - 输出：HDMI 视频输出、GPIO 控制信号（蜂鸣器驱动）、USB 供电输出。
（2）摄像头接口模块
系统支持三路摄像头接入方案： - USB 摄像头：通过 USB 3.0 接口即插即用，兼容 UVC 协议； - MIPI CSI 摄像头：通过 15Pin FPC 排线连接，支持 NV12 原始帧输出； - GS130W 摄像头：通过专用接口接入，适用于低照度环境。
关键信号线：USB 差分信号线（D+/D-）、MIPI CSI 时钟线（CLK+/CLK-）与数据线（D0+/D0- ~ D3+/D3-）。
（3）告警驱动模块
蜂鸣器驱动：RDK X5 GPIO 输出 PWM 信号经三极管放大后驱动有源蜂鸣器，低电平触发。
通信：通过 UART 串口（TX/RX）与 RDK X5 连接，支持 AT 指令控制，摔倒触发时自动拨打预设号码。
（4）电源管理模块
采用 5V/4A 稳压电源适配器为主控板供电，避免 GSM 发射时的电流跌落干扰主控。电源输入端加入 TVS 二极管与滤波电容，提升系统抗干扰能力。

2.3.1 软件整体介绍

•前端展示层：基于 HTML5 + JavaScript 的单页应用（SPA），集成 PWA 能力，提供实时视频查看、告警记录、系统仪表盘、AI 对话、心情日志及设置页面。

2.3.2 软件各模块介绍
（1）YOLO 检测与 MJPEG 推流模块（fall_detection.py）
该模块为系统核心，负责视频采集、BPU 推理及视频流推送。
# 关键函数流程示意
def main_loop():
    cap = init_camera()          # 初始化摄像头（USB/MIPI/GS130W 自适应）
    model = load_bpu_model()      # 加载 YOLO11n-pose INT8 量化模型
    while running:
        frame = cap.read()        # 读取 NV12 帧
        results = model.infer(frame)  # BPU 推理：输出人体框 + 17 关键点
        fall_flag = detect_fall(results)  # 三级过滤摔倒判定
        if fall_flag:
            trigger_alerts()      # 触发多渠道告警
        mjpeg_server.push(frame)  # 推送 MJPEG 流至 8080 端口
        hdmi_display.show(frame)  # HDMI 本地显示
关键输入变量：frame（NV12 格式图像帧）、model_path（BPU 模型路径）。 关键输出变量：results（检测框坐标、关键点坐标、置信度）、fall_flag（布尔型摔倒判定结果）。
（2）三级过滤摔倒检测模块
def detect_fall(results):
    # 第一级：框形状过滤
    aspect_ratio = box_width / box_height
    if aspect_ratio < 1.3: return False
    
    # 第二级：垂直速度过滤
    vertical_speed = calc_vertical_velocity(keypoints_history)
    if vertical_speed < SPEED_THRESHOLD: return False
    
    # 第三级：姿态角度过滤
    body_angle = calc_body_angle(keypoints)  # 基于 17 个关键点
    if body_angle > ANGLE_THRESHOLD: return False
    
    # 持久化过滤：连续 N 帧确认
    return persistent_confirm(fall_buffer)
关键输入：keypoints（17×2 关键点坐标数组）、keypoints_history（时序关键点缓存）。 关键输出：fall_flag（摔倒判定布尔值）。
（3）多渠道告警模块
告警调度器采用异步事件驱动架构，支持并行触发： - alert_buzzer()：GPIO 控制蜂鸣器鸣响 5 秒； - alert_wechat()：调用 Server 酱 API，携带 SendKey 推送微信消息； - alert_voice_call()：调用腾讯云语音电话 API； - alert_web()：通过 SSE 向 Web 前端推送告警事件。
（4）表情识别模块（emotion_recognizer.py）
基于面部关键点提取 ROI 区域，输入轻量级表情分类模型，输出 7 类情绪概率分布。关键输入：face_roi（面部裁剪区域）；关键输出：emotion_label（情绪类别）、confidence（置信度）。
（5）Flask Web 服务端（app.py）
提供以下核心路由： - /：系统主页与仪表盘； - /camera：摄像头实时查看页（直连 8080 端口 MJPEG 流）； - /alerts：告警记录查询页（读取 alerts.json）； - /chat：AI 对话页（代理转发至 DeepSeek API）； - /mood：心情日志页（汇总表情识别历史）； - /api/status：系统状态 RESTful API； - /api/config：用户配置读写接口。

系统软件已完成全部功能开发，主要界面包括： - 仪表盘首页：实时显示系统运行状态、BPU 占用率、今日告警统计； - 摄像头监控页：嵌入 MJPEG 实时视频流，叠加人体骨骼关键点与检测框； - 告警记录页：按时间倒序展示历史告警事件，支持详情查看与导出； - AI 对话页：集成 DeepSeek 对话接口，支持自然语言交互； - 心情日志页：可视化展示表情识别统计结果； - 设置页：配置摄像头参数、告警阈值、通知渠道开关。

第四部分 总结
4.1 可扩展之处
1.多摄像头轮巡与全景覆盖：当前系统支持单路摄像头接入，未来可通过 USB Hub 或 IP 摄像头扩展至多路视频源，实现客厅、卧室、卫生间等多区域全覆盖，并在 Web 端提供分屏轮巡视图。
2.夜视红外与低照度增强：针对夜间或弱光环境下的看护需求，可集成红外补光灯与低照度增强算法，或更换为支持 IR-CUT 的摄像头模组，实现 24 小时全天候可靠监测。
3.语音呼救关键词检测：在现有视觉检测基础上，增加音频通道的”救命”“帮帮我”等关键词识别，构建视觉+听觉的双通道确认机制，进一步降低漏报风险。
4.事件视频自动截取与日报生成：摔倒事件发生前后 5 秒视频片段自动保存，并基于每日监测数据生成 PDF 看护日报，包含活动时长、情绪分布、告警事件等统计信息，便于家属与医生回溯分析。
5.云端模型持续学习：建立边缘端异常样本回传机制，在云端对误检、漏检样本进行模型微调与迭代更新，通过 OTA 方式推送至边缘设备，实现系统的持续进化。
4.2 心得体会
本作品的研发历时数月，从最初仅能实现简单人体框检测的 YOLOv5s 原型，逐步演进为融合姿态估计、表情识别、多渠道告警与 Web 交互的完整系统，过程中经历了多次技术路线的调整与关键难题的攻克，收获颇丰。
在模型选型与部署阶段，我们最初采用”YOLOv5s 检测 + 独立姿态模型”的级联方案，但在 RDK X5 上运行时帧率不足 3 FPS，且内存占用接近上限。经过深入调研，我们转向 YOLO11n-pose 单模型方案，并通过地平线工具链完成 INT8 量化，最终将帧率提升至 8 FPS，同时释放了约 30% 的内存空间。这一经历让我们深刻认识到，在边缘算力受限场景下，模型架构的轻量化设计与硬件特性的深度适配同等重要。
在摔倒检测算法设计阶段，早期仅依赖检测框宽高比的判定策略在测试中产生了大量误报——例如老人弯腰系鞋带、坐下休息等正常动作均被误判为摔倒。为解决这一问题，我们引入了基于 17 个关键点的身体倾角计算与垂直速度分析，并设计了”三选二”投票与连续帧确认机制。经过数十组场景视频的迭代调参，误报率从初期的 15% 降至 3.2%。这一过程让我们体会到，鲁棒的算法不仅需要理论支撑，更离不开真实场景数据的反复验证。
在视频流传输优化阶段，我们曾尝试通过 Flask 代理 MJPEG 视频流，但很快发现单线程模型下视频流会阻塞 API 请求，导致告警推送与状态查询严重延迟。经过架构重构，我们改为浏览器直连摄像头 HTTP 服务器的方案，Flask 仅负责控制信令与数据 API，从根本上解决了并发冲突问题。这一教训让我们铭记：Web 架构设计必须充分考虑 IO 模型与并发特性，不能简单套用传统方案。
在系统整合与稳定性测试阶段，我们发现有模块在拨打电话时的大电流脉冲会导致 RDK X5 重启。通过为 GSM 模块增加独立供电与电源隔离，并优化 UART 通信的流控机制，最终解决了这一硬件稳定性问题。这让我们意识到，软硬件协同设计中的电源管理与电磁兼容问题不容忽视。
回顾整个研发历程，我们不仅掌握了边缘 AI 部署、量化推理、Web 全栈开发等硬核技术，更深刻理解了”以用户为中心”的产品思维——技术方案的价值最终要通过解决真实痛点来体现。当看到系统在测试环境中准确识别摔倒动作并秒级触发告警时，我们真切感受到了技术赋能社会关怀的力量。未来，我们将继续打磨这一作品，期待它能为更多独居老人与儿童的家庭带去安心与守护。
