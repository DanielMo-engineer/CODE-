# 2026-07-07 — RDK X5 综合平台搭建

## 项目：fall_ai_web（端口 5050）

今天从零搭建了一个完整的 RDK X5 综合平台，包含：
- **主页** — 卡片入口（摄像头 / AI 对话 / 对讲机 / 设置）
- **摄像头页** — 实时 MJPEG 画面 + 状态覆盖（人数/FPS/报警弹窗）
- **AI 对话页** — QQ 风格聊天界面，支持语音输入（Web Speech API）+ 自动朗读回复（TTS）
- **对讲机页** — 长按说话，松开发送，服务器中转广播
- **设置页** — 配置 DeepSeek API Key、摄像头地址、测试连接、网络信息

## 关键技术决策

### 摄像头代理问题
- 摄像头服务器（8080）的 HTTP 服务器是**单线程**的（HTTPServer而非ThreadingHTTPServer）
- Flask 通过 socket 代理 `/stream` 会占住唯一线程，导致 `/api/status` 轮询全部超时
- **最终方案**：浏览器**直连**摄像头服务器，不经过 Flask 代理
  - MJPEG 流：`<img src="http://IP:8080/stream">`
  - 状态轮询：`fetch(http://IP:8080/api/status)` 每秒一次
  - 路由：前端先通过 `/api/settings` 获取 CAMERA_URL，再拼接 `/stream` 和 `/api/status`
- 平台状态 `/api/status` 返回的总是 OFFLINE 因为 poll_fall_status 后台线程被单线程阻塞

### MJPEG 代理边界问题
- 摄像头 MJPEG 边界是 `fr`（标记为 `--fr`）
- 之前的代理硬编码 `boundary=--boundary` 导致浏览器解析失败
- 修复：用 socket 直连获取源服务器 HTTP 响应头，解析 Content-Type 提取真实 boundary

### 双端口方案（已废弃）
- 尝试了 HTTP(5050) + HTTPS(5051) 双端口运行
- 尝试了 `make_server`、`subprocess`、`threading` 等多种方式
- **最终方案**：只用 HTTP(5050) 单端口，HTTPS 由 cpolar 隧道提供

### PWA 离线支持
- Service Worker 缓存应用外壳（/ /camera /chat /offline）
- 离线页面显示"设备已离线"
- API 请求失败时返回离线状态 JSON
- manifest.json 支持"添加到主屏幕"
